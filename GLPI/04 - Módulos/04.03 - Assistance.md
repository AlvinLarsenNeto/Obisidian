## Matriz de Prioridade

O GLPI calcula prioridade automaticamente com base em **Urgência × Impacto**:

| Urgência ↓ / Impacto → | Muito alto | Alto | Médio | Baixo | Muito baixo |
|---|---|---|---|---|---|
| Muito alto | Muito alta | Muito alta | Alta | Média | Média |
| Alto | Muito alta | Alta | Alta | Média | Baixa |
| Médio | Alta | Alta | Média | Baixa | Baixa |
| Baixo | Média | Média | Baixa | Baixa | Muito baixa |
| Muito baixo | Média | Baixa | Baixa | Muito baixa | Muito baixa |

Valores numéricos: Very High=5, High=4, Medium=3, Low=2, Very Low=1.  
Fórmula: `priority = round((urgency + impact) / 2)` com alguns ajustes nos extremos.

## SLA / OLA / Contratos de nível de serviço

### Conceitos
- **SLA (Service Level Agreement)**: contrato com o cliente — mede tempo de resposta e resolução do ponto de vista externo.
- **OLA (Operational Level Agreement)**: acordo interno entre equipes — mede tempo de resposta e resolução internamente.
- **Time to own**: tempo até o chamado ser assumido por um técnico (primeira atribuição).
- **Time to resolve**: tempo até o chamado ser resolvido/fechado.

### Como o GLPI calcula o SLA
1. O SLA é vinculado ao chamado via **regras automáticas** (Assignment rules) ou manualmente.
2. O GLPI define um **prazo** (deadline) somando o tempo configurado no SLA ao horário de abertura do chamado.
3. O cálculo respeita o **calendário de trabalho** (business hours) configurado no SLA — apenas horas úteis contam.
4. Se o ticket não for resolvido antes do deadline → SLA **breached** (violado).

### Fórmula exata — SLA de Resposta (TTO)
Confirmado via `glpi-debug-sla` contra MySQL direto em 2026-04-21:

```sql
-- DENOMINADOR: tickets com time_to_own IS NOT NULL (TTO SLA configurado)
-- OK:
takeintoaccountdate <= time_to_own
-- BREACHED:
takeintoaccountdate > time_to_own
-- EXCLUÍDO da base:
time_to_own IS NULL
-- NÃO-OK (conta no denominador mas não no numerador):
takeintoaccountdate IS NULL  (ticket nunca foi assumido)
```

**Observação crítica:** O GLPI preenche `takeintoaccountdate = date` (data de abertura) automaticamente em muitos tickets via regras/automação. Esses tickets passam na comparação `date <= time_to_own` (sempre verdadeiro) e são contados como 'ok'. **Não são falsos positivos** — o GLPI considera o SLA de resposta cumprido quando o sistema registra a atribuição, mesmo que automática.

### Fórmula exata — SLA de Solução (TTR)
```sql
-- OK (considera sla_waiting_duration = tempo que ticket ficou pendente, pausando o SLA):
solvedate <= CASE
  WHEN sla_waiting_duration > 0
  THEN DATE_ADD(time_to_resolve, INTERVAL sla_waiting_duration SECOND)
  ELSE time_to_resolve
END
-- BREACHED: condição inversa
-- EXCLUÍDO: time_to_resolve IS NULL ou solvedate IS NULL
```

**`sla_waiting_duration`**: quando o ticket entra em status "pending" (aguardando cliente), o GLPI pausa o contador de SLA. Esse tempo acumulado é somado ao prazo original antes de verificar se houve violação.

### Campos relevantes no banco (`glpi_tickets_cache` / GLPI MySQL)
| Campo MySQL | Campo Cache | Descrição |
|---|---|---|
| `date` | `opened_at` | Data/hora de abertura |
| `time_to_resolve` | `time_to_resolve` | Prazo calculado para resolução (com calendário) |
| `solvedate` | `solved_at` | Data/hora efetiva de resolução |
| `time_to_own` | — (não cacheado) | Prazo para atribuição (TTO deadline) |

### Status de SLA no Portal ASV
Calculado em `src/lib/slaMetrics.ts`:
- `ok` — resolvido antes do prazo
- `at_risk` — aberto e dentro do prazo mas próximo do vencimento
- `breached` — prazo ultrapassado

`formatSlaRate` **trunca** (não arredonda) — 99,99% nunca vira 100%.

### Calendários de trabalho
Configurados em **Configuration > Service Levels**. Definem quais horas/dias contam para o SLA.  
Exemplo: SLA de 4h em horário comercial (8h–18h, seg–sex) = prazo efetivo diferente de 4h corridas.

## Tickets

### Tipos
- **Incident**: algo que parou de funcionar (impacto operacional).
- **Request**: solicitação de serviço (novo acesso, novo equipamento, etc.).

### Ciclo de vida (status)
`New → Processing (assigned) → Processing (planned) → Pending → Solved → Closed`

No Portal ASV mapeados como: `new | open | pending | waiting_client | resolved | closed`

### Campos importantes
- `entities_id` — entidade (empresa) à qual o ticket pertence
- `users_id_recipient` — quem abriu
- `groups_id_assign` — grupo atribuído
- `users_id_assign` — técnico atribuído (no GLPI pode ser múltiplos via tabela `glpi_tickets_users`)
- `assigned_to` no cache ASV — texto com nome completo (join via `glpi_users_cache.full_name`)

### Apontamentos (Planning / Task)
- Tabela `glpi_tickettasks` no MySQL GLPI
- Cache ASV: `glpi_planning_cache`
- `actiontime` em **segundos** (dividir por 60 para comparar com `technician_goals.daily_goal_minutes`)
- Usar `task_date` (não `begin_time`) para filtrar por período

## Problemas (Problems)
Agrupa incidentes recorrentes com a mesma causa raiz. Um problema pode estar vinculado a múltiplos tickets.

## Mudanças (Changes)
Processo formal para alterações na infraestrutura. Pode ter aprovações, tarefas e impactos vinculados.

## Tickets recorrentes
Permite criar templates de tickets que se repetem automaticamente em periodicidade configurável.
